# 與 Claude Code 的協作紀錄

> 紀錄本專案從零到一是如何透過與 [Claude Code](https://claude.com/claude-code) 對話協作完成的。
> 目的不是炫技，而是分享一個「用 AI 當搭檔開發」的真實流程，給對 AI 輔助開發有興趣的人參考。

---

## TL;DR

| 項目 | 說明 |
| --- | --- |
| 主要工具 | Claude Code（CLI 形式） |
| 協作模式 | 自然語言下指令 → AI 產生程式碼 → 我審視 / 提出修改 |
| 對話輪數 | 約 6 ~ 8 輪即完成 v0.3.0 |
| 我的角色 | 規劃需求、設計風格方向、審視輸出、決定下一步 |
| AI 的角色 | 落實程式碼、處理樣式細節、確保 RWD、寫初版內容 |

## 協作哲學

我把 Claude Code 當成一位**很懂前端但需要被引導的工程師夥伴**。
這意味著：

1. **需求由我定義** — AI 不會憑空知道我要的「科技感」是哪一種「科技感」，我要先想清楚再說。
2. **內容由我審視** — AI 寫出的描述、技術細節、措辭都需要我檢查，避免出現我實際沒做過的事。
3. **架構由我決定** — 例如「技能要不要有獨立詳細頁？」這種影響檔案結構的決策由我提出。
4. **AI 負責落地** — 一旦方向確定，由 AI 處理 CSS 細節、JS 動態渲染邏輯、RWD 斷點等耗時但機械的工作。

## 開發歷程

### Round 1 — 初版骨架

我給的需求：

> 「請幫我建立一個個人線上履歷網站，使用純 HTML、CSS、JavaScript，不要使用框架。風格做成科技感、乾淨、適合實習面試展示。」

附上我的個人資料（姓名、背景、技能、3 個專案）。

Claude Code 輸出：

- `index.html`、`style.css`、`script.js` 三個檔案
- Hero、Skills（8 張）、Projects（3 張）、Contact 區段
- 預設深色配色 + 青綠強調色（`#64ffda`）
- 打字機效果、Canvas 粒子背景、RWD

**我做的事**：確認資料無誤、檢查視覺方向是否符合預期。

### Round 2 — 本機預覽

我下了一句：

> `cd /home/user/tuna && python3 -m http.server 8000`

Claude Code 啟動了背景 HTTP server，並引導我用 `http://localhost:8000` 開啟。

**學到的事**：在 sandbox / 遠端環境工作時，要記得告訴對方 port forwarding 怎麼設。

### Round 3 — 技能子頁

我提出新需求：

> 「技術卡可以幫我做成點進去會到新的頁面介紹該技能嗎？並有返回鍵可以回到主頁。」

Claude Code 提出兩個方案：
- (A) 為每個技能做獨立 HTML（8 個檔案）
- (B) 用一個 `skill.html` 模板 + URL `?id=` 動態渲染

我採用方案 B。產出：

- 新增 `skill.html`
- `script.js` 新增 `SKILLS` 資料物件與渲染邏輯
- `index.html` 將每張技能卡改為 `<a>` 連結
- 加入「← 返回首頁」按鈕

**我審視**：每個技能的描述是否真的反映我會的東西，不會的就請對方改寫。

### Round 4 — 專案子頁

提出類似需求：

> 「project 也可以幫我加入按鈕並進入更詳細的頁面去介紹嗎？」

Claude Code 沿用相同模式：

- 新增 `project.html`
- `script.js` 新增 `PROJECTS` 資料物件，含「概述／動機／系統流程／技術挑戰／技術棧／成果」六段結構
- `index.html` 為每個專案加上「查看完整介紹 →」按鈕

**面試取向的設計**：「技術挑戰與解法」是專門為面試問答設計的段落，預期面試官會深入問。

### Round 5 — 文件化

也就是現在你正在看的這份。

需求：

> 「請幫我新增 CLAUDE.md / SKILL.md / README.md / TODO.md / CHANGELOG.md，內容像可以放到 GitHub 專案展示。」

## 我學到的協作技巧

### ✅ 有效的做法

1. **一次只推一個議題** — 不要一口氣丟 5 個需求，分多次對話可讓 AI 在每個議題上更聚焦。
2. **先給範圍再給細節** — 「我想要科技感深色風」比「背景請設 #0a0e1a」更能讓 AI 發揮。
3. **明確指出風格參考** — 若提到「類似某設計師的作品集風格」會比抽象描述更精準。
4. **要求列出選項** — 當我不確定怎麼做時，請 AI 先給 2~3 個方案。
5. **拒絕不對的內容** — 不要因為 AI 寫得「看起來很厲害」就照單全收，所有描述都要是真的我做過的事。

### ❌ 沒這麼有效的做法

1. **過度抽象的指令** — 「做得好看一點」這種沒幫助。
2. **跳過審視** — 把 AI 寫的內容直接複製貼到履歷，可能會出現自己其實不會的技術。
3. **忽略架構決策** — 把所有東西都丟一個檔案會難維護，要主動提出拆分需求。

## 結語

整個流程下來，我大約花了 **不到一個下午** 就完成了一個有完整 RWD、多頁面、有動態效果的個人履歷網站。

但更重要的是：在這個過程中，我不是「被 AI 取代」，而是「**用 AI 把自己的能力放大**」。
我依然需要懂 HTML / CSS / JavaScript 才能：

- 判斷 AI 寫的程式碼是不是垃圾
- 知道某個 CSS 改動會不會破壞 RWD
- 看出某段 JS 邏輯有沒有 bug
- 決定整體架構走向

**AI 是放大鏡，不是望遠鏡。它放大你既有的能力，但看不到你不知道的東西。**
